# apps/desktop/src/views/SettingsView.layout.css

Геометрия экрана настроек после редизайна раскладки (ветка t5-settings-space,
2026-07-16): колонки во всю высоту зоны `<main>`, текучие ширины с потолками
читабельности, скролл внутри панели. Перекрывает габариты базового каркаса
`.muza-settings*` из `app.css` селекторами повышенной специфичности.
Импортируется из `SettingsView.tsx`. Создано: 2026-07-16.

---

## Почему отдельный файл, а не правка app.css
`app.css` в момент правки параллельно редактировали другие ветки — брифом
задано «править app.css минимально и аддитивно». Оверрайды отсюда мержатся
без конфликтов; в `app.css` добавлен только маркер-комментарий в конце файла.
Следствие: числа в комментариях app.css (220px, max-width 680) — УСТАРЕВШИЕ
для габаритов, но механика (`@container`, стекло, типографика рельса) живёт
по-прежнему там. Комментарии в самом layout.css подробнее этой выжимки.

## Механика (и почему именно так)
- **Высота — цепочка процентов, НЕ 100vh**: `.muza-view:has(> .muza-settings)
  { height: 100% }` → `.muza-settings { height: 100% }` → `__cols { flex: 1;
  min-height: 0; grid-template-rows: minmax(0, 1fr) }`. 100vh нельзя: у корня
  Player может стоять `zoom` (prefs.uiScale, App.tsx ~1394), а vh внутри
  зумированного поддерева меряет реальный вьюпорт — вылезало бы за окно.
  Без поддержки `:has` цепочка тихо деградирует к прежнему «по высоте
  контента» (percentage от auto-родителя = auto) — грациозный фолбэк.
  ⚠️ `:has` матчит только ПРЯМОГО ребёнка — внутрипанельные div'ы с классом
  `muza-view` (анимация смены вкладки, `paneClass`) не задевает.
- **Скролл переехал с `<main>` внутрь `.muza-settings__pane`** (`overflow-y:
  auto`, скроллбар спрятан как у main). `min-height: 0` на обоих грид-итемах
  обязателен — иначе контентный минимум раздувает ряд выше зоны. Сброс
  скролла при смене раздела — эффект в SettingsView.tsx (`paneScrollRef`).
- **Рельс**: `grid-template-columns: clamp(220px, 24cqw, 280px) minmax(0,1fr)`.
  220 — старый минимум (подписи читаемы), 280 — дефолт `--w-sidebar` (рельс
  не солиднее главного рельса приложения), 24cqw — плавный рост (280
  достигается к ~1170px ширины контейнера ≈ окно ~1500). cqw меряет контейнер
  `settings` — ту же величину, что и `@container`-порог.
- **Панель**: `max-width: 880px` (потолок читабельности строк; прежние 680 и
  были «пустыней справа») + `width: 100%` + `margin-inline: auto` — на
  сверхшироком окне контент дрейфует к центру СВОЕЙ зоны, при треке ≤880
  margin'ы нулевые и панель сидит вплотную к рельсу, как раньше.
- **Иконочный рельс 48px** — тот же порог `@container settings (max-width:
  600px)`; колонки переобъявлены здесь потому, что базовое правило (0,2,0)
  иначе перебивало бы правило app.css из контейнер-запроса (0,1,0).
  Скрытие подписей/заголовка и сужение гэпа по-прежнему в app.css.
- **`border-radius: var(--r-lg)`** на рельсе — полновысотная колонка
  скругляется как зоны (сайдбар/панели), карточный r-md оставлял бы «плашку».
- **`flex-shrink: 0` на пунктах и заголовке рельса** — метрика 48px это
  инвариант (как NavItem сайдбара); при нехватке высоты рельс скроллится
  (`overflow-y: auto`), а не сжимает пункты.
- **`@media (max-height: 719px)` прячет `__title`** — вертикальный аналог
  скрытия в узком рельсе; `aria-label` на tablist держит имя экрана. Расчёт:
  контент рельса ≈ 582px; зона = окно − 92 (плеербар) − 3×12 (гэпы) →
  заголовок влезает от ~710px окна; на минимальных 700 (tauri.conf.json)
  без него остаётся ~44px запаса. @media по ВЫСОТЕ честен (в отличие от
  ширины): вертикаль зоны не зависит от перетаскиваемых боковых панелей.
  ⚠️ Числа завязаны на токены `--h-playerbar: 92` / `--gap-zone: 12`
  (packages/ui/src/tokens/spacing.css) и `--fs-h1: 1.75rem` — поменяются
  токены, пересчитай порог (страховка overflow-y всё равно не даст наездов).
- **Паддинги**: `.muza-settings { padding: 0 var(--sp-6) 0 0 }` — рельс
  прижат к верху/низу/левому краю зоны (параллелен сайдбару, тот занимает
  весь свой грид-ряд); верхний/нижний отступ содержимого панели — в
  `paneStyle` (SettingsView.tsx), внутри скроллера.

## Контракт с TSX
Селекторы с `>` завязаны на вложенность `.muza-settings > .muza-settings__cols
> (nav.muza-settings-nav + .muza-settings__pane)` — её сторожит
`SettingsView.layout.test.tsx`. Переименование классов/обёртки без правки
этого файла молча убьёт раскладку.
