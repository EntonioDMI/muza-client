# apps/desktop/src/views/SettingsView.tsx

Экран «Настройки»: все разделы (`settings.tabs.*`, вертикальная навигация слева +
панель справа) и под-экраны (customize/equalizer/discord/market/data/sessions/
privacy/licenses/bar/nav/stats — см. `i18n/en.ts` за списком зон `settings.*`).
Большой файл (~3720 строк), несколько компонентов-секций внутри, каждый со своим
`const { t } = useT();` (и местами `lang`). Полностью переведён в T29/T30 (i18n
эпик W5) — эта заметка НЕ покрывает файл целиком, только зафиксированное далее.
Обновлено: 2026-07-15 (вертикальная навигация, заголовок внутрь плашки).

---

`type T = (key: TranslationKey, params?: TParams) => string;` (строка ~51) —
тип функции перевода, передаётся параметром в свободные module-level
хелперы без доступа к React-контексту (напр. `discordPreviewVars(t: T)`) —
паттерн для переиспользования `t()` вне компонента.

**T34a (2026-07-14, эпик W5, общий свип):** `plugins/install.ts` получил
параметр `lang: Lang` в `pickAndStagePlugin`/`stagePluginFromMarket` (раньше
эти функции сами бросали захардкоженные русские `Error` и задавали текст
нативного диалога выбора файла). Оба места вызова здесь обновлены передать
свой `lang` (уже был в `const { t, lang } = useT();`, строка ~848):
- `startInstall()` (около строки 871): `pickAndStagePlugin(lang)`.
- маркетплейс-путь установки (около строки 1366):
  `stagePluginFromMarket({ manifest, code, css, strings }, lang)`.

Без этой правки установка плохого файла плагина показывала бы русский тост
об ошибке даже в EN-интерфейсе (текст ошибки шёл из `e.message`, который
`catch`-блок предпочитает переведённому фолбэку —
`e instanceof Error ? e.message : t(...)`).

Известный документированный трейд-офф (НЕ тронуто в T34a): `EQ_PRESETS`
(около строки 810) и литерал `"Свой"` (кастомный пресет эквалайзера,
строка ~1435/2099) — персистентные значения `Prefs`, общая схема с
`apps/web`; перевод сломал бы совместимость сохранённых профилей. См.
`docs/knowledge/packages/app/src/i18n/en.ts.md` (словарь уехал в `@muza/app`
в Э0 веб-паритета, 2026-07-15 — на старом пути остался пенёк).

## Вертикальная навигация вместо `<Tabs wrap>` (коммит `73b05bb`, 2026-07-15)
Бэкфилл: аргументация жила только в сообщении коммита и в комментариях
`apps/desktop/src/app.css` (строки 41-226 — **там подробнее, чем здесь**: замеры,
пороги, отброшенные варианты; при правке вида читай их, а не эту выжимку).

- **Причина**: `<Tabs wrap>` переносился по ДЛИНЕ ПОДПИСЕЙ (`flex: 1 0 auto` →
  точка переноса = функция текста): RU давал 5+5, EN — 7+3, `flex-grow` раздувал
  недобитый ряд. Вертикальный список от длины подписей не зависит по построению.
- **Состав**: `SETTINGS_TAB_KEYS` (~713) + `SETTINGS_TAB_ICONS` (~733) +
  `SettingsNav` (~753, локальный компонент). Рендер: `.muza-settings` →
  `.muza-settings__cols` → `<SettingsNav>` + `.muza-settings__pane` (~3345-3360).
- **`role="tablist"`, а не ссылки**: выбор раздела мгновенно меняет соседнюю
  панель и никуда не «уходит» (ни маршрута, ни истории). Плюс к ролям `Tabs` из
  `@muza/ui` — `aria-orientation="vertical"` и связка `aria-controls` ↔
  `aria-labelledby` (`navItemId(key)`, `SETTINGS_PANE_ID`, строки 746-748).
  Roving tabindex со стрелками намеренно нет.
- **`aria-label` + `title` на пункте всегда**, даже когда подпись видна: узкая
  панель прячет `.muza-settings-nav__label` СТИЛЕМ (`@container`), а не условным
  рендером — `@container` из JS не виден.
- **Активный пункт — по `aria-selected`**, стилем из CSS (без JS-замера пилюли).
- **Смена раздела закрывает под-экран** (`setSub(null)` перед `setTab`); под-экран
  рендерится в правую колонку, навигация остаётся с подсвеченным разделом.
- ⚠️ Схлопывание в иконочный рельс — **container query по ширине самой панели**
  (`@container settings (max-width: 600px)`), не `@media` по окну: боковые зоны
  приложения пользователь тянет руками, `window.innerWidth` тут врёт.

## Дизайн настроек: стекло плашки + заголовок внутрь неё (2026-07-15)
Правки по жалобам владельца, проверены живьём замерами computed-style. В ЭТОМ
файле изменилось мало — почти всё в `app.css`:

- **Заголовок «Настройки» переехал ВНУТРЬ плашки навигации**: старый инлайн-стилевой
  `<h1>` — сосед `.muza-settings__cols` снаружи (~3336) — удалён; теперь `<h1
  className="muza-settings-nav__title">` — первый ребёнок `<nav>` внутри
  `SettingsNav`. Причина: снаружи заголовок не совпадал по левому краю ни с чем
  (текст 24px, иконка пункта 40px, подпись 72px — три разных края). После: один
  край на 348px.
- **`<h1>` и `aria-label` у `role="tablist"` намеренно дублируют одну строку**
  (`t("settings.title")`): в узком рельсе CSS прячет `<h1>` (`display:none`), а
  `aria-label` от ширины не зависит — скринридер заголовок не теряет.
- В `app.css` (не здесь): `.muza-settings-nav` получила
  `background: var(--glass-sidebar, var(--surface-1))` + `backdrop-filter:
  var(--bf-zone, none)` — была единственной поверхностью приложения вне системы
  стекла; `--glass-sidebar` переиспользован сознательно (тот же навигационный
  рельс на том же `--bg-0`), свой ключ prefs не заводился. Метрики пункта
  сравнялись с NavItem сайдбара: 40→48px, `--sp-3`→`--sp-4`.
- Проверено в EN и RU: навигация 220 и панель 676 идентичны — инвариант `73b05bb`
  держится.
